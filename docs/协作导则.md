# 协作导则 · 镜面反射装置

> 给参与本项目的协作者的约束规则。
> 配合《项目档案》阅读。目的：让协作者快速理解红线，在不破坏核心设定的前提下提供帮助。

---

## 〇、最高原则

**这是一个持续 15 年以上的长期作品。协作者的角色是协助实现与拓展，不是重新设计。**
不确定时：保留已确立的设定，对开放问题提供可能性而非定论。

---

## 一、绝不能改错的硬设定（HARD CONSTRAINTS）

构成装置根基的事实。修改它们等于做了另一个装置：

1. **镜面倾角 45° → 光路旋转 90°**。装置的"宇宙常数"。不要改成其他角度作为默认（除非明确要求探索变角）。

2. **红 / 绿 / 蓝 = 同一个世界内部的三个层级，不是三个独立的世界**。
   - 共享同一个 Σ + F，在 Σ 视点处合成为一幅完整画面。
   - ❌ 不要写成 "three worlds"、"红色世界/绿色世界/蓝色世界"。
   - ✅ 正确表述："一个世界的三个层级 / 三个活动域（strata / habitable zones）"。
   - 这是早期已纠正过的常见错误，必须牢记。

3. **"世界"的计数单位是 Σ，不是层**。一个 Σ + F + 它生成的全部层级 = 一个完整世界。变体二有 2 个世界（正/背），变体三有 2 个世界（左/右对称），变体四有 N 个世界——每个世界各自内含红绿蓝三层。

4. **层间相似放大**：W₁（最内/最近）最小，越往外越大，最外层镜面 M₃ 极大。不要做成等大或反向。

5. **镜面尺寸有限 → bug 区域**：光路超出镜面物理边界即丢失。世界不是无限大的。这个有限性是装置的本质（对应"穿帮/坠入黑暗"），不要为了视觉好看而无限延展。

6. **Σ 是理想化的全知造物主视点**，不是有限/受难的神。"焦距对不准"是技术担忧，不是设定。不要把它哲学化为"神的局限"。

---

## 二、必须保持开放的问题（DO NOT RESOLVE）

以下问题保持开放（未定）。可以探讨、列可能性，但**禁止拍板、写死、或在代码/文档里固化为唯一答案**：

1. **变体二中正背两世界的"人"是否相同**（复制人生 / 完全不同 / 互补纠缠）——开放。
2. **变体五中心的 X 空间如何定义**——开放。
3. **收缩动画的叙事顺序**（哪一层先消失）——开放（取决于如何叙事，未定）。代码里应做成**可配置**，不要硬编码一种顺序。
4. **生成新世界的机制**——"触碰生成"这类人本/事件驱动的设定已否决；方向是"规律驱动"，但尚未选定。不要自作主张填入任何机制。
5. **是否引入曲率 / 四维 / 反射回路**——都是构思中的方向，不要擅自实现进主线。可留扩展接口。

> 判断法则：如果一个设定在《项目档案》"开放方向"一节里，就是开放的。

---

## 三、关于"生成新世界机制"的特别说明

设定红线：**拒绝生硬的、人本主义的、巧合驱动的、事件触发式的设定**（例如"两个世界的人同时触碰镜面就生成第三个世界"——已弃用，太套路）。

方向是：**符合客观规律、让人觉得理所应当、自然涌现**的机制。
候选（仅供参考，未定）：临界相变、对称破缺、驻波节点、不动点定理、共形边界条件等"规律驱动而非意志驱动"的机制。

➡️ 在此问题定论前，任何涉及"世界如何生成/相交/影响彼此"的实现，都应**保持中性、可参数化、不绑定特定叙事机制**。

---

## 四、沟通与工作方式（STYLE RULES）

1. **不要过度工程化、不要过早堆实现细节**。概念探索阶段就跟着探索，不要急着给工程方案；会明示何时进入实现。

2. **不要用"列 5 个方案 A/B/C/D/E"的方式轰炸**正在畅想的话题。畅想时就一起畅想。

3. **不要强行给开放问题"完美收口"**。容纳歧义是本项目的工作方式，过早闭合不受欢迎。

4. **几何必须精确**。对反射方向、比例、对称极其敏感。涉及几何实现时，先做最小可验证单元（如"单面镜+单层反射"）确认方向对了再扩展。反射方向错了后面全错。

5. **对外叙事聚焦几何与体验本身**。展览文案、对外描述均围绕几何与概念展开，不引入创作背景。

6. **交流平实、精确、对等**；不做心理分析、不过度共情、不灌鸡汤。

7. **可以用专业术语**（参数化、本体论、对称群、反射矩阵等），不必过度解释基础概念。中英文皆可。

8. **纠正要被真正吸收**。理解被纠正后就更新它，不要表面承认后继续用旧理解。

---

## 五、技术实现约束（针对 web / 代码协作）

1. **逻辑重写，非文件转换**。不要尝试调用 Rhino / Rhino.Compute / 任何 .NET 依赖。原 .gh 是纯几何变换（Mirror/Rotate/Move/Surface），Three.js 原生可实现。

2. **既定实现路线 = 视锥追迹（frustum ray tracing）**。不要回退到"反射矩阵 + 整体缩放"的简化方案。当前代码已确立：从 Σ 发出穿过 F 四角的四条光线，逐面反射追迹；**镜面 = 视锥在该位置的截面**（尺寸由光锥决定，不预设）；层级放大是视锥发散的自然结果。这是经过几何验证的正确路线，不要推翻。

3. **必须系统化，不是单页堆砌**。15 年长期项目，不能停留在"一个几百行的 index.html 把所有逻辑堆在一起"。已模块化为：几何内核（CORE，纯数学）/ 场景渲染（RENDER）/ 参数状态（APP）/ UI / 动画 分离。新增功能（变体、动画、收缩、反射）应作为可插拔模块加入，不是往单文件里继续塞代码。

4. **不要用 localStorage / sessionStorage 或任何浏览器存储 API**（若嵌入某些沙箱会失败）。状态用内存；如需持久化用可选的导入/导出 JSON（预设文件），不依赖浏览器存储。

5. **架构要易扩展**。会持续加新维度（曲率、4D、反射回路，对应预留的未激活参数）。把"参数 + 时间 → 几何"做成清晰可复用的纯函数模块（输入参数对象，输出几何描述，与渲染解耦）。

6. **保留扩展接口**：原 .gh 滑块 16–19 是预留未激活的扩展维度（当前值为 0）。代码里可为未来方向留钩子，但不要现在就实现进去。

7. **核心参数（必须可调）**：镜面倾角(默认45)、层数(1–3)、每面镜面间距、旋转复制 N、层间放大率、F 的位置与尺寸。其余比例系数可先固定。**参数定义应集中在一处（single source of truth），UI 从参数定义自动生成，不要手写重复的滑块代码。**

8. **颜色语义固定**：红=底层/最内，绿=中层，蓝=高层/最外。不要随意改配色语义。

9. **几何与渲染解耦**：几何计算（视锥追迹、活动域、可布景体）必须是不依赖 Three.js 的纯数学模块（只用向量运算），这样未来能在 Node 端跑测试、能复用、能换渲染后端。Three.js 只负责把几何描述画出来。

10. **可能的远期集成**：Blackplant（contract-first 时空数据平台）。若架构能把"参数+时间→几何"做成类似 parametric_definition 的可复用单元，利于未来集成。非当前要求，但架构方向应朝此对齐。

## 六、一句话总结给协作者

> 严守 45°、三层属一世界、Σ 是全知视点、镜面有限边界这几条硬设定；
> 对正背双生、X 空间、收缩顺序、生成机制这些开放问题只提供可能性、不替项目定论；
> 拒绝生硬的人本/事件驱动设定，偏好规律涌现；
> 几何要精确、沟通要平实、架构要可扩展；
> 你在协助一个活了 15 年、还会继续生长的作品。
