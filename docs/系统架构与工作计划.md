# 系统架构与工作计划 · 镜面反射装置 Web

> 架构指令 + 分阶段工作计划。
> 目标：把当前的单文件 `index.html`（几何正确但一次性）重构为**可持续生长 15 年的模块化系统**。
> 配套：《项目档案》《协作导则》。

---

## A. 为什么要重构（现状诊断）

**当前状态**：`web/index.html`，约 430 行，所有逻辑（几何计算 / 渲染 / 参数 / GUI / 动画）堆在一个 `<script>` 里。

**优点**：几何正确（视锥追迹、45°、穿帮、放大全部验证通过）、能跑、零构建依赖。
**问题**：
- 参数定义散落（`params` 对象 + 手写的每个 `gui.add(...)` 重复劳动）。
- 几何与 Three.js 渲染耦合（`buildSingleWorld` 里既算几何又建 mesh），无法单独测试几何、无法复用。
- 加新变体 / 新动画只能往单文件里继续塞，会膨胀成不可维护的巨石。
- 没有任何测试，几何回归错误无法自动发现。

**这与项目"15 年持续生长"的本质冲突**。必须现在重构，趁代码还只有 430 行。

> 重构原则：**不推翻已验证正确的几何逻辑（视锥追迹），只是把它从单文件里拆出来、模块化、加测试**。重构后视觉行为应与现在**完全一致**（这是验收标准：重构不改变任何可见效果）。

## B. 目标架构

### 核心理念：三层解耦

```
┌─────────────────────────────────────────────┐
│  CORE（几何内核）— 纯数学，零 Three.js 依赖     │  ← 可在 Node 测试 / 可复用到 Blackplant
│  输入: 参数对象  →  输出: 几何描述(纯数据)      │
├─────────────────────────────────────────────┤
│  RENDER（渲染层）— Three.js                    │  ← 把几何描述画成 mesh/line/sprite
│  输入: 几何描述  →  输出: THREE.Group          │
├─────────────────────────────────────────────┤
│  APP（应用层）— 状态 / UI / 动画 / 交互         │  ← 编排：参数变 → 重算 → 重绘
└─────────────────────────────────────────────┘
```

**关键纪律**：CORE 绝不 `import three`。它只接收数字、输出 `{points:[...], quads:[...], beams:[...]}` 这样的纯数据。这样：
- 几何能在 Node 里跑单元测试（验证 45°、放大比、穿帮，回归不出错）。
- 几何能复用到别处（未来 Blackplant 集成、或换 Babylon/WebGPU 渲染）。
- 维护者改几何逻辑时不碰渲染，改外观时不碰几何。

### 建议目录结构

```
web/
├─ index.html                  # 仅入口：挂载点 + importmap + 一句 import './src/main.js'
├─ vendor/                     # 保持现状（three / lil-gui / addons，已下载好）
├─ src/
│  ├─ main.js                  # 应用启动：装配 core→render→app
│  │
│  ├─ core/                    # ── 几何内核（纯数学，禁止 import three）──
│  │  ├─ vec.js                #   最小向量运算（add/sub/dot/cross/norm/planeBasis…）
│  │  ├─ mirror.js             #   mirrorNormal / reflectDir / rayPlane
│  │  ├─ frustum.js            #   视锥追迹核心：参数 → 各层镜面四角 + 光路 + 穿帮层 + 九宫格投影
│  │  ├─ activity.js           #   ★安全区(黄)/穿帮区(红) + 任意平面截面 + 层地面切分 + 布景锚点
│  │  └─ terrain.js            #   地形：全局连续噪声场（展开坐标/冻结带/首层贴地）+ 两遍构造编排
│  │
│  ├─ render/                  # ── Three.js 渲染层 ──
│  │  ├─ scene.js              #   renderer/scene/三相机/controls/天空环境(equirect)/太阳/resize
│  │  ├─ builders.js           #   几何描述 → mesh: Mirror/Reflector/Floor/Ground/Terrain/Figure/Wall/Scenery/Label
│  │  ├─ materials.js          #   共享配色常量（黄安全区/红穿帮区，单一来源）
│  │  └─ worldView.js          #   几何描述(纯数据) → THREE.Group
│  │
│  ├─ app/                     # ── 应用层 ──
│  │  ├─ params.js             #   ★ 参数 schema：单一数据源（含范围/默认/分组/说明）
│  │  ├─ gui.js                #   从 params schema 自动生成 lil-gui 面板
│  │  ├─ animator.js           #   展开/收缩/旋转生成 的时间驱动（含可配置收缩顺序）
│  │  └─ store.js              #   状态：当前参数 + onChange 订阅 + 预设导入/导出(JSON)
│  │
│  └─ variants/               # ── 变体作为可插拔模块（未来生长点）──
│     ├─ README.md            #   如何新增一个变体
│     ├─ base.js              #   变体一/三：当前的单/多世界（已实现，迁移到此）
│     ├─ semitransparent.js   #   变体二：半透明正背双世界（待做，留空壳+接口）
│     └─ compound.js          #   变体五：复合生成序列（待做，留空壳+接口）
│
├─ test/                       # ── 几何回归测试（Node 运行，无需浏览器）──
│  ├─ mirror.test.mjs          #   45°→90°、反射方向
│  ├─ frustum.test.mjs         #   三层放大递增、穿帮触发条件
│  └─ run.mjs                  #   极简测试 runner（不引入 jest，零依赖）
│
├─ docs/
│  ├─ 项目档案.md
│  ├─ 协作导则.md
│  └─ 系统架构与工作计划.md      # 本文件
│
└─ presets/                    # 参数预设（替代浏览器存储做持久化）
   ├─ default.json
   ├─ variant3-C3.json         # 三世界 C₃
   └─ author-real-scale.json   # .gh 实测尺度
```

> 注：保持**零构建工具**（继续用 importmap + ES modules，不引入 Vite/webpack/npm install），除非维护者明确要求。模块拆分用原生 `import`/`export` 即可。这样维护者本地双击 / 简单静态服务器就能跑，符合维护者偏好（Ubuntu/LNMP 本地部署）。

## D. 几何描述格式（CORE 与 RENDER 之间的契约）

CORE 输出、RENDER 消费的中间数据结构，建议形如：

```javascript
// core/world.js 的输出（纯数据，无 THREE 对象）
{
  seed:    { pos:[0,0,0], radius: 0.13 },
  frame:   { corners: [[x,y,z],...4] },
  mirrors: [ { layer:0, color:'red',   corners:[[...]×4] },
             { layer:1, color:'green', corners:[[...]×4] },
             { layer:2, color:'blue',  corners:[[...]×4] } ],
  beam:    [[x,y,z], ...],            // 中心光路折线
  frustum: [ [[...],...], ×4 ],        // 四条角光线折线
  safe:    [ { gap:0, points:[[...]] }, ... ],  // ★黄·安全可布景区 (本锥 减 前/后邻锥)
  bug:     [ { gap:0, points:[[...]] }, ... ],  // ★红·易穿帮区 (本锥 ∩ 邻锥左右侧壁; 布景物落入即穿帮)
  labels:  [ { text:'1.0', pos:[...], color:'red' }, ... ],
  bugLayer: 0,                         // 0=无穿帮, n=第n层穿帮
}
```

RENDER 层的 `worldView.js` 只读这个对象 → 产出 `THREE.Group`。**任何"它长什么样"都在 RENDER；任何"它是什么形状"都在 CORE。**

## E. 工作计划（分阶段，每阶段可独立验收）

> 总原则：**每个阶段结束都要能跑、能让维护者看效果、不破坏已验证的几何**。先重构（不改外观），后加功能。

### 阶段 0 · 测试护栏（先做，~半天）
**目的**：在重构前先锁住"正确"，重构后用它验证没改坏。
- [ ] 写 `test/run.mjs`（30 行的零依赖 runner：跑 *.test.mjs，断言失败就报错）。
- [ ] `mirror.test.mjs`：断言 45° 镜面把 (1,0,0) 反射成 (0,0,-1)，夹角 90°。
- [ ] `frustum.test.mjs`：用当前默认参数，断言三层对角尺寸 **递增**（≈3.13 < 5.18 < 8.79）；断言把某层间距设到极端值会触发 `bugLayer>0`。
- **验收**：`node test/run.mjs` 全绿。

### 阶段 1 · 几何内核抽离（核心重构，~1–2 天）
**目的**：把几何从 index.html 里搬进 `core/`，**纯数学、零 three**，外观不变。
- [ ] `core/vec.js`：把现在依赖 THREE.Vector3 的运算改写成纯函数（或保留一个极薄的向量类）。
- [ ] `core/mirror.js` `core/frustum.js` `core/activity.js`：迁移 `mirrorNormal/reflectDir/rayPlane / 视锥追迹 / activityRegion(安全区·黄) + 易穿帮区(红·本锥∩邻锥侧壁) / buildSingleWorld 的几何部分`，输出 D 节的纯数据结构（**剥掉所有 `new THREE.Mesh` 等渲染代码**）。
- [ ] `core/world.js`：单世界几何 + 多世界旋转复制（纯数据）。
- [ ] 让阶段 0 的测试改为 import 这些新模块——**必须仍全绿**。
- **验收**：测试全绿；几何逻辑已无 three 依赖。

### 阶段 2 · 渲染层抽离（~1 天）
- [ ] `render/scene.js` `materials.js` `builders.js` `worldView.js`：迁移所有 `new THREE.*`。
- [ ] `worldView.js` 消费 CORE 的纯数据 → THREE.Group。
- [ ] `main.js` 串起来：core 算 → render 画。
- **验收**：浏览器里的画面与重构前**像素级一致**（这是关键验收点：重构不改变任何可见效果）。

### 阶段 3 · 参数 Schema + 自动 UI（~1 天）
- [ ] `app/params.js`：写 C 节的 schema（含 .gh 编号与"推断/待核对"标注）。
- [ ] `app/gui.js`：遍历 schema 自动生成 lil-gui（删掉手写的重复 `gui.add`）。
- [ ] `app/store.js`：状态 + onChange→rebuild 订阅 + 预设 JSON 导入/导出按钮。
- [ ] `presets/`：导出当前默认值 + .gh 实测尺度 + C₃ 三世界 三个预设。
- **验收**：改 schema 一处即可增删参数；能导出/载入预设 JSON。

### 阶段 4 · 变体模块化（~1 天）
- [ ] `variants/base.js`：把现有单/多世界逻辑收纳为"变体一/三"模块（注册式：每个变体导出 `{id, label, buildGeometry(params)}`）。
- [ ] `variants/semitransparent.js` `compound.js`：建空壳 + 接口 + README，**不实现内容**（留给维护者定方向）。
- [ ] UI 加"变体选择"下拉。
- **验收**：能在变体一/三间切换；新增变体只需加一个文件并注册。

### 阶段 5 · 动画系统（~1–2 天）
- [ ] `app/animator.js`：时间驱动的参数插值。
- [ ] **真实降维收缩**：面(2D)→线(1D)→点(0D) 的逐层降维（不是淡入淡出）。
- [ ] **收缩顺序可配置**：`outer-first`（蓝先撤，维护者当前演示顺序）/ `inner-first` / `simultaneous` / `custom`。**默认 outer-first，但绝不写死，维护者会换叙事**。
- [ ] **多世界生成过渡**：N 从 1→2→3 平滑长出（"一生二三"）。
- [ ] 镜面边界发光描边（bug 区域边界更清晰）。
- **验收**：能播放展开/收缩，且收缩顺序可切换。

### 阶段 6 · 真实镜面反射（可选，性能允许时）
- [ ] 用 `THREE.Reflector` 或 render target 让镜面里真的映出其他层。
- [ ] 注意多次反射的性能；可加开关、可降采样。
- [ ] 为"反射回路"（最外层反射回内层）留接口——但**机制本身未定，不要擅自实现**，只搭可关闭的实验开关。
- **验收**：镜面呈现真实反射；不卡。

### 永续 · 扩展位（不排期，维护者驱动）
曲率 / 极小曲面、四维投影、反射回路反馈、Blackplant 集成——都作为新模块加入，对应预留参数。每次加都遵守"CORE 纯数学 / RENDER 画 / 变体即插件"的纪律。

## F. 工程纪律（硬约束）

1. **零构建优先**：继续 importmap + ESM，不引入 npm/Vite/webpack（除非维护者要）。模块用原生 import/export。
2. **CORE 禁止 import three**。违反此条 = 架构失败。
3. **重构阶段（1–3）不改变任何可见效果**。验收靠"画面一致 + 测试全绿"。
4. **每阶段独立可跑、可验收**。不要一次性大爆改后交一个跑不起来的东西。
5. **每阶段做完先验收，再进下一阶段**。几何需精确，早发现早纠正。
6. **不碰开放设定**：收缩顺序、生成机制、正背双生含义、X空间——只搭可配置接口，不写死答案（见《协作导则》二、三节）。
7. **注释保留中文/中英混合**，维护者两者都行。
8. **遇到需要维护者拍板的几何参数**（如 mDist 真实拆分、frameH 宽高比），不要猜死——在 UI 和注释里标"★推断, 待维护者核对"，并在该阶段交付时列成"待确认清单"。

## G. 待确认清单（实现前先确认）
